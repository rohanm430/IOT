1. Ensure that multi-factor authentication is enabled for all privileged users   [State: Not Applicable]  [Priority: High] 

Category:Spoofing
Description:Enable multi-factor authentication for all user credentials who have write access to Azure resources.
Justification:MFA can be enabled, not really a concern for the labs
Possible Mitigation(s):Enable multi-factor authentication for all Privileged user credentials Refer: https://tinyurl.com/astm-identity
SDL Phase:Design
Policy:Audit accounts with owner permissions who are not MFA enabled on a subscription
Documentation:https://docs.microsoft.com/en-us/azure/active-directory/authentication/howto-mfa-userstates
Implemented Mitigation(s):
Steps:Enable remember Multi-Factor Authentication Sign in to the Azure portal. On the left, select Azure Active Directory > Users and groups > All users. Select Multi-Factor Authentication. Under Multi-Factor Authentication, select service settings. On the Service Settings page, manage remember multi-factor authentication, select the Allow users to remember multi-factor authentication on devices they trust option. Set the number of days to allow trusted devices to bypass two-step verification. The default is 14 days. Select Save.

2. An adversary may gain unauthorized access to Data Lake Analytics account in a subscription  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:An adversary may gain unauthorized access to Data Lake Analytics account in a subscription
Justification:Accessit based on Role based access for users. We use Azure Managed identities to run the output data from Stream Analytics to Data Lake Storage
Possible Mitigation(s):Enable Role-Based Access Control (RBAC) to Azure storage account using Azure Resource Manager. Refer: https://aka.ms/tmtauthz#rbac-azure-manager
SDL Phase:Implementation
Policy:
Documentation:https://azure.microsoft.com/documentation/articles/storage-security-guide/#management-plane-security
Implemented Mitigation(s):
Steps:When you create a new storage account, you select a deployment model of Classic or Azure Resource Manager. The Classic model of creating resources in Azure only allows all-or-nothing access to the subscription, and in turn, the storage account. With the Azure Resource Manager model, you put the storage account in a resource group and control access to the management plane of that specific storage account using Azure Active Directory. For example, you can give specific users the ability to access the storage account keys, while other users can view information about the storage account, but cannot access the storage account keys.

4. An adversary may execute unknown code on IoT Device  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:An adversary may launch malicious code into IoT Device and execute it
Justification:Will use TLS to make the connection secure and reliable. Will use Security Token for secure access control.
Possible Mitigation(s):Ensure that unknown code cannot execute on devices. Refer: https://aka.ms/tmtconfigmgmt#unknown-exe
SDL Phase:Design
Policy:
Documentation:https://docs.microsoft.com/windows/iot-core/secure-your-device/securebootandbitlocker
Implemented Mitigation(s):
Steps:UEFI Secure Boot restricts the system to only allow execution of binaries signed by a specified authority. This feature prevents unknown code from being executed on the platform and potentially weakening the security posture of it. Enable UEFI Secure Boot and restrict the list of certificate authorities that are trusted for signing code. Sign all code that is deployed on the device using one of the trusted authorities.
